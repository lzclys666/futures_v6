## 三、详细冲突分析

### 🔴 C1: YIYI 时间竞争（高风险）

**问题描述**：
- 系统架构 W1 任务 1.3：CSV路径统一检查（YIYI负责，P1）
- 因子升级 Week 1：完整的数据审计（YIYI负责，核心任务）

**冲突分析**：
- 系统架构的 CSV 路径检查是**交易闭环的前置条件**
- 因子升级的数据审计是**整个升级计划的基础**
- 两者都需要 YIYI 投入大量时间

**解决方案**：
```
建议调整：
1. 系统架构 W1 任务 1.3（CSV路径检查）提前到 4.24-4.25（2天完成）
2. 因子升级 Phase 0 数据审计从 4.26 开始
3. 或者：将 CSV 路径检查交给 mimo 执行，YIYI 只负责验收
```

---

### 🟡 C2: 前端资源竞争（中风险）

**问题描述**：
- 系统架构 W2-W3：持仓看板开发（Lucy负责）
- 因子升级 Week 4：三图联动 Dashboard（前端负责）

**冲突分析**：
- 两个任务都需要前端开发资源
- 但时间上有 1-2 周间隔，可错峰

**解决方案**：
```
建议调整：
1. 持仓看板（系统架构）优先，因为是交易闭环的必要组件
2. 因子升级 Dashboard 可延后到 Week 5，或与持仓看板复用组件
3. 两个 Dashboard 使用相同技术栈（React + ECharts），可复用基础组件
```

---

### 🟡 C3: CSV→Parquet 迁移时间差（中风险）

**问题描述**：
- 因子升级 Week 2：启动 CSV→Parquet 双写迁移
- 系统架构 W6-W7：性能优化阶段才做 CSV→Parquet

**冲突分析**：
- 因子升级提前做 Parquet 是为了统计模块查询性能
- 系统架构后期做 Parquet 是为了整体性能优化
- **不冲突，反而互补**：因子升级先验证 Parquet 可行性，系统架构后期全面推广

**建议**：
```
两个路线图保持一致：
1. 因子升级 Week 2 启动双写（兼容现有CSV）
2. 系统架构 W6-W7 基于因子升级的经验，全面迁移到 Parquet
3. 共享 Parquet Schema 和迁移脚本
```

---

### 🟢 C4/C5: 风控与 Regime 检测（互补，低风险）

**分析**：
- 系统架构的风控规则（13条）是**交易执行层**的保护机制
- 因子升级的失效预警是**信号生成层**的质量监控
- 两者互补：失效预警发现因子问题 → 风控规则阻止错误交易

**宏观熔断与 HMM Regime**：
- HMM Regime 检测（因子升级 Week 3）提供"因子是否失效"的判断
- 宏观熔断机制（系统架构 W6-W7）基于多个因子的 Regime 状态决策是否暂停交易
- **建议**：HMM Regime 输出作为宏观熔断的输入信号之一
