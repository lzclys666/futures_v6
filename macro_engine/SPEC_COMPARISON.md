## 一、架构层对比

| 维度 | 我的方案 | 用户方案 | 谁更完整 |
|------|---------|---------|---------|
| **视角** | 数据质量 + 元数据治理 | 运行时可靠性 + 故障恢复 | 互补 |
| **层数** | L1-L4（4层） | L0-L5（6层） | 用户多 L0+L5 |
| **L4 定义** | IC 有效性 | run_all 集成测试 | 完全不同（需合并）|

**核心差异：我的 L4 = 用户不存在的「IC 每日计算」，用户的 L4 = 我完全没有的「run_all 全链路」。**
**→ 两个 L4 都需要保留，合并为 L4（IC）+ L5（run_all）。**

## 二、我的方案缺口（14 项）

### G1: 缺少 L0 文件系统预检
**用户方案有：** `requests.get` 残留扫描、`__pycache__` 清理、`.py.old` 文件检查
**我的方案：** 完全无此层
**补充内容：**
- L0.1: `grep -rl "requests.get" crawlers/ --include="*.py" | wc -l` → 期望 0
- L0.2: `find crawlers/ -name "*.py.old"` → 期望 0
- L0.3: 验证 fetch_url 覆盖文件数 ≥24

### G2: 缺少 fetch_url 返回值解包验证
**用户方案有：** 检查 `(data, err) = fetch_url()` 解包模式
**我的方案：** 仅检查 `fetch_url in content`
**危险：** 有脚本可能写成 `html = fetch_url(url)`（错误），语法层面不报错，运行时才失败
**补充内容：** L1.3 增加返回值解包模式检查（正则匹配 `(data|html|text|err|r)\s*,\s*(err|error)`）

### G3: 缺少 L2 运行时快速验证
**用户方案有：** 每个品种挑 1 个因子脚本实际运行（10 个 P0 + 10 个 P1 + 2 个跨品种）
**我的方案：** 无运行时验证
**补充内容：** P0 品种（AU/CU/RB/AG/RU）× 5，P1 品种 × 10，实际执行脚本，验证 DB 写入

### G4: 缺少 L4 run_all 集成测试
**用户方案有：** 运行 `AU_run_all.py`、`CU_run_all.py` 等，验证子因子串联采集
**我的方案：** 无
**补充内容：** 5 个 P0 run_all + 10 个 P1 run_all，验证 stderr 无 ERROR，所有子因子打印 `[OK]`

### G5: 缺少 L5 降级逻辑验证
**用户方案有：** 模拟 L1 失败，验证 L2/L3/L4 降级链（L1→L2→L3→L4→DB回补）
**我的方案：** 无降级链设计
**问题：** 规范中有「网络错误不 raise」，但没有降级机制设计
**补充内容：** 降级层级定义（L1 网络→L2 缓存→L3 历史均值→L4 跳过）

### G6: 缺少 P0/P1 品种分级
**用户方案有：** P0（AU/CU/RB/AG/RU）、P1（J/AL/NI/TA/HC）、P2（其余 20 个品种）
**我的方案：** 无分级，笼统处理
**补充内容：** 三级品种体系 + 对应验收标准

### G7: 缺少 `ic_heatmap` 表名不匹配问题
**用户方案 SQL：** `FROM factor_data`
**实际情况：** 表名是 `pit_factor_observations`（另有 `ic_heatmap` 独立表）
**补充内容：** 所有 SQL 使用正确表名 `pit_factor_observations`

### G8: 缺少 `factor_metadata` 表同步验证
**实际情况：** `factor_metadata` 表仅 19 行，应有 239 行
**我的方案：** 提到但无验证步骤
**补充内容：** L3.4 增加 `SELECT COUNT(*) FROM factor_metadata` → 应 ≥239

### G9: 缺少 ic_heatmap 覆盖范围验证
**用户方案：** 无关注（只检查表是否存在）
**我的方案：** 提到但未量化
**补充内容：** ic_heatmap 应覆盖全 34 品种（当前仅 5 个）

### G10: 缺少降级链三层设计（架构缺陷）
**用户方案 L5：** 降级验证，但没有定义降级顺序
**我的方案：** 完全没有降级机制
**补充内容：**
```
降级链：
  L1 → 原始 fetch_url 失败
  L2 → 备用 URL（两个以上数据源）
  L3 → 最近一次缓存值（mem/redis）
  L4 → 历史均值填充（7日EWMA）
  L5 → 标记为 null，不阻断（因子可空）
```

### G11: 缺少品种因子健康度评分量化标准
**我的方案：** 有 health_score() 函数，但未与验收方案整合
**补充内容：** 80 分健康 / 60-80 警告 / <60 不健康，三档阈值写入验收报告

### G12: 缺少 B3-6.2 web_utils 自定义 headers 完整性测试
**用户方案 L5.2：** 验证 Sina/金十/Eastmoney/Mysteel/铝道网 5 个源
**我的方案：** 完全未提
**补充内容：** 5 个自定义 headers 源的连通性测试（已在用户方案中完整定义）

### G13: 缺少 YAML-DB 差异处理时间表
**我的方案：** 列出 116+62 差异，但无处理时间表
**补充内容：** 优先级排序 + 截止日期

### G14: 缺少脚本化验收报告模板
**用户方案：** 有完整的 markdown 报告模板
**我的方案：** 无报告模板
**补充内容：** 采用用户方案的报告模板（比我定义的更完整）
