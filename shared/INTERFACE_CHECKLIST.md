# 接口工作清单

> **版本**: v1.0  
> **日期**: 2026-05-01  
> **目的**: 任何涉及跨模块接口修改前，强制检查此清单  

---

## 规则 1：修改前必须验证

### Python schemas.py（后端 API）

改之前必须做：
```
1. curl http://localhost:8000/api/macro/signal/RU
2. 确认你要改的字段确实存在（或不存在）
3. 只改真实存在的差异
4. 禁止新增"预期有但 API 不返回"的字段
```

### TypeScript types/*.ts（前端类型）

改之前必须做：
```
1. 找 deep 要真实 API 响应 JSON 示例
2. 或 curl http://localhost:8000/api/macro/signal/RU 自己看
3. 确认字段名、类型、可选性与 API 一致
4. 禁止擅自添加/删除 API 不返回的字段
```

### CSV 格式（数据文件）

改之前必须做：
```
1. 找 YIYI 确认当前 CSV 所有消费者
2. 确认改动影响范围
3. 在 decisions_log 记录变更提案
```

紧急修复（48 小时内）可以先改后通知，但必须：
- 在 decisions_log 说明"紧急修复，原因"
- 24 小时内通知 owner

## 规则 3：phantom 字段零容忍

**定义**：phantom 字段 = 代码/类型中定义了，但 API 实际不返回的字段。

**处理方式**：
- 发现 phantom → 立即删除
- 不确定是否存在 → 先验证，再修改

**常见 phantom 来源**：
- 预期 API 有但实际没有的字段（这次 deep 的 `signal`/`strength`）
- API 改了但旧类型定义没清理
- 从其他项目复制来的字段但本项目不用

---

## 规则 4：跨模块修改必须经过 decisions_log

**触发条件**：修改他人负责的接口文件，或修改会影响其他模块的共享格式。

**标准流程**：
```
1. 在 decisions_log.md 新增记录：
   ### [日期] [模块A] 拟修改 [接口名]
   - 改什么：...
   - 为什么：...
   - 影响范围：...
   
2. 等 owner 回复确认（或 24 小时内无异议视为同意）

3. 动手修改

4. 在同一记录下补充：
   - 实际改动内容
   - 验证结果
```

## 检查清单（修改前必填）

改任何接口文件前，对照检查：

```
[ ] 我看过真实 API 响应（或找 owner 要了示例）
[ ] 我确认要改的字段存在/不存在
[ ] 我没有新增 phantom 字段
[ ] 如果改的是他人负责的文件：我已通知 owner 并得到确认
[ ] 如果改动影响跨模块：我已在 decisions_log 记录
[ ] 我知道这个改动会触发哪些联动修改，已通知相关人
```

## 典型错误对照

| 错误做法 | 正确做法 |
|---------|---------|
| "我觉得 API 应该有这个字段" → 直接加上 | 先 curl 看实际响应 |
| "TS 类型里没这个字段" → 直接删 Python | 先问 deep 要 API JSON |
| "这个配置文件随便改" | 先确认 owner，通知相关人 |
| "改完再说" | 先记录，owner 确认，再改 |
