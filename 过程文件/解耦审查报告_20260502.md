## 第二部分：模块接口耦合分析

### 🔴 严重问题

#### C1. 风控规则四套不兼容定义（P0）

| 方面 | routes/risk.py | macro_api_server.py | PaperBridge | core/risk_engine.py |
|------|----------------|-------------------|-------------|-------------------|
| R10 阈值 | score<-0.5 禁多 >0.7 禁空 | "-0.5"(字符串) | -0.5 vs 0.45 | fuse_low=30, fuse_high=70 (0-100制) |
| Layer | layer1/2/3 | L1~L5(5层) | 数字1/2/3(3层) | 注释1/2/3(3层) |
| 严重级别 | PASS/LOW/MEDIUM/HIGH | normal/warning/triggered | PASS | PASS/WARN/BLOCK |
| 规则数 | 12 (含R12) | 9 | 12 (含R12) | 11 (无R12) |

**后果**: 同一笔下单请求，不同接口给出不同的风控结果。前端不知道该信任哪个数据源。

> **2026-05-03 更新**：⚠️ **部分修复**。`routes/risk.py` 和 `vnpy_bridge.py` 的 layer 格式已从字符串 `"layer1"/"layer2"/"layer3"` 统一为数字 `1/2/3`，API `/api/risk/status` 返回 Int32 类型。但 `core/risk/rules_config.yaml` 单一真相源文件尚未创建（B1-4 交付物），其他 3 套定义（阈值/severity/规则数）仍未统一。

#### C2. bridge 双注入路径（P0）

- `api/routes/trading.py` → `from services.vnpy_bridge import bridge`（路径A）
- `api/routes/risk.py` → `from services.vnpy_bridge import bridge`（路径A）
- `macro_api_server.py` → 先尝试路径A，若 bridge.status != "running" 则返回 None
- `run.py` → 通过 `set_vnpy_bridge(vnpy_bridge)` 注入（路径B）

**两套路径可能指向不同实例或无实例**。如果 `run.py` 在 `vnpy_bridge.start()` 前调用 `set_vnpy_bridge()`，而 routes 同时从模块级导入 `bridge`，两者可能不同。

> **2026-05-03 范围更新**：实际影响范围扩大至 9 处，除上述 4 处外还包括 `api/routes/position.py:39`、`api/routes/vnpy.py:9`（**模块级导入**，启动即触发），以及 `api/macro_api_server.py:74`。其中 5 处为模块级导入，影响最严重的是 `vnpy.py` 和 `trading.py`。

#### C3. routes/risk.py 自行实现风控（P1）

`api/routes/risk.py` 的 `/api/risk/simulate` 端点完全绕过了 `core/risk/risk_engine.py`：
- 不使用 `RiskRule` 抽象基类
- 不使用 `RiskEngine.check_order()` 规则链
- 使用 `bridge.is_trading_hours()`, `bridge.get_account()` 等硬编码的桥接器接口
- 风控逻辑散布在 300 行函数中，难以维护和验证

这是一个典型的 **"复制而不是复用"** 反模式。

### 🟠 中等问题

#### C4. FactorDetail 模型字段名不一致

```
api/schemas.py:          normalized_score  -> alias "normalizedScore"
api/macro_api_server.py: factor_value      -> alias "factorValue"  ← 不同
```

两个文件定义了不同的 FactorDetail 模型，但 macro_api_server.py 中的模型从未被使用（该文件直接返回 dict 而非序列化模型），`schemas.py` 的模型也未被任何路由使用。

#### C5. CSV 字段映射不一致

```
api/macro_scoring_engine.py:  factor_direction -> contributionPolarity  (factor固有方向)
services/signal_bridge.py:    直接读 contributionPolarity               (贡献度极性)
```

`factor_direction`（因子固有方向属性：positive/negative/neutral）和 `contributionPolarity`（贡献度极性，由factor_value×weight的符号决定）不是同一概念。映射混用可能导致信号方向判断错误。

#### C6. 前端 API 数据契约不一致

```
前端 types/risk.ts:
  R3_TOTAL_POSITION, R8_DRAWDOWN, R9_CONCENTRATION, R11_DISPOSITION

后端 routes/risk.py:
  R3_PRICE_LIMIT, R8_TRADING_HOURS, R9_CAPITAL_SUFFICIENCY, R11_DISPOSITION_EFFECT
```

规则 ID 的完整名称 3/4 不一致，前端无法正确映射后端返回的风控规则。

> **2026-05-03 更新**：✅ **部分修复**。规则 ID 字符串已对齐（`R3_PRICE_LIMIT`/`R8_TRADING_HOURS`/`R9_CAPITAL_SUFFICIENCY`/`R11_DISPOSITION_EFFECT`），前端 TypeScript 类型已与后端一致。**遗留**：`types/risk.ts:81` 的 `RiskSeverity` 枚举仍为 `'PASS' | 'LOW' | 'MEDIUM' | 'HIGH'`，未同步为后端的 `'PASS' | 'WARN' | 'BLOCK'`（影响风控级别颜色/图标渲染）。

#### C7. strategies 直接依赖 core/risk

`strategies/macro_risk_strategy.py` 直接 `from core.risk.risk_engine import RiskEngine`
这违反了分层架构中 strategies 应只依赖 VNpy CtaTemplate 的原则。

#### C8. core/risk 存在两个副本

- `D:\futures_v6\core\risk\risk_engine.py` — 真正实现
- `D:\futures_v6\risk\risk_engine.py` — 重复副本，测试文件导入的是这个副本

`tests/test_risk_engine.py` 中的 `from risk.risk_engine import` 导入的是根目录副本，不是 core 中的正式版本。

> **2026-05-03 更新**：✅ **已修复**。根目录 `risk/risk_engine.py` 副本已删除，`tests/test_risk_engine.py` 引用已修正为 `core.risk.risk_engine`。
