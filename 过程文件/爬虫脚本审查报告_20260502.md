## 二、跨品种共享因子目录说明

### 背景：CU_NI、ZN_PB_SN、_shared 不是期货品种

经源码核实，以下三个目录**不是独立期货品种**，而是**跨品种共享因子目录**：

| 目录 | 性质 | 服务品种 | 用途说明 |
|------|------|---------|---------|
| `CU_NI/` | 跨品种共享因子 | CU（沪铜）、NI（沪镍） | LME 铜/镍升贴水，两个品种共同受影响的国际现货价差因子 |
| `ZN_PB_SN/` | 跨品种共享因子 | ZN（沪锌）、PB（沪铅）、SN（沪锡） | LME 锌/铅/锡 3M 跨月价差，三个有色金属共同因子 |
| `_shared/` | 通用工具模块 | 全部品种 | 公共因子基类、数据校验、通用爬虫工具 |

**设计意图正确**：这些目录不需要独立的 `run_all.py`，而是**被相关品种的 run_all 脚本调用**。

## 三、🔴 严重问题

### H1. CU_run_all.py 跨目录硬编码调用 — 耦合方式脆弱

- **问题**：`CU_run_all.py` 第 22 行硬编码 `../CU_NI/CU_NI_抓取LME升贴水_EVENT.py`
- **风险**：目录结构变动即断裂，依赖相对路径跳级，不符合模块化设计
- **建议**：改为通过公共模块注册表或配置化映射调用，例如 `_shared/cross_symbol_factors.yaml` 中声明 `"CU": ["CU_NI/LME升贴水"]`，由统一的跨品种调度器执行

### H2. ZN_PB_SN/fetch_lme_zn_pb_sn.py 脚本质量不合格

- 文件名应为 `ZN_PB_SN_LME升贴水.py`，现在是英文名
- 文件头标记「待修复」，`FACTOR_CODE`/`SYMBOL` 未定义
- **数据写入 CSV 而非 pit_data.db**，无 `save_to_db()` 调用
- 无 L1/L2/L3/L4 降级逻辑
- **3 处硬编码 CSV 路径**：
  - `D:\futures_v6\macro_engine\data\crawlers\ZN\daily\ZN_LME_3M.csv`
  - `D:\futures_v6\macro_engine\data\crawlers\PB\daily\PB_LME_3M.csv`
  - `D:\futures_v6\macro_engine\data\crawlers\SN\daily\SN_LME_3M.csv`
- **影响**：ZN、PB、SN 三个品种的 run_all 若调用此脚本，数据不入库仅写 CSV，宏观打分引擎读取不到，因子链断裂

### H3. CU_NI 脚本自建 get_pit_date() 与公共模块重复

`CU_NI_抓取LME升贴水_EVENT.py` 自建了日期回退函数，与 `common.db_utils.get_pit_dates()` 功能重复，且周一回退逻辑不一致（公共模块回退 3 天到上周五，自建只回退 2 天到周六），可能导致节后第一天数据偏差。

**建议**：CU_NI 脚本应复用 `_shared` 或 `common` 模块的公共函数，保持跨品种因子与主品种因子的日期逻辑一致。

### M1. `except: pass` 静默丢弃异常（4 处最严重）

| 文件 | 行 | 问题 |

| `AU/AU_SPDR黄金ETF持仓量.py` | 85 | `except:\n    pass` |

| `JM/JM_计算焦煤期现基差.py` | 56 | `except:\n    pass` |

这些完全静默了异常，脚本失败时不会打印任何信息，排障全靠猜。`AU_SPDR黄金ETF持仓量.py` 虽然在 L1/L2 层有 `except Exception as e` 打印，但在 L1c 层内部嵌套了一个裸 `except: pass`，导致该分支的异常被静默。

> **2026-05-03 更新**：✅ **已修复**。全局 `grep -rn "except: pass"` 零命中，4 处静默异常已全部清除。

### M2. 裸 except 捕获（13 个文件，21 处）

以 `AO_run_all.py` 为例：
```python
try:
    subprocess.run([sys.executable, script])
except:  # ← 应改为 except Exception as e
    print(f"失败: {e}")
```
不指定异常类型会误抓 `KeyboardInterrupt`、`SystemExit`，导致用户无法 Ctrl+C 退出。

**涉及文件清单**：AO_run_all.py、AU_SPDR黄金ETF持仓量.py、BU_沥青期货净持仓.py、EC_run_all.py、J_焦炭期货近远月价差.py、JM_计算焦煤月差.py、JM_计算焦煤期现基差.py、LC_run_all.py、LH_run_all.py、NI_run_all.py、P_run_all.py、P_批次2_月报爬虫.py、_shared/exchange/collect_exchange_data.py

> **2026-05-03 更新**：✅ **已修复**。全局裸 `except:` 零命中，13 个文件的 21 处裸 except 已全部改为 `except Exception as e`。

### M3. DEVNULL 抑制子进程输出（AU、CU、I）

`AU_run_all.py`、`CU_run_all.py` 和 `I_run_all.py` 的子进程调用中：
```python
subprocess.run([...], stdout=subprocess.DEVNULL, stderr=subprocess.DEVNULL)
```
子爬虫的所有控制台输出（包括错误信息）被完全丢弃。如果某个因子采集失败，run_all 无法给出任何有效诊断信息。

## 六、四级降级模式评估

| 级别 | 说明 | 实现情况 |
|------|------|---------|
| L1 | 主数据源直接采集 | ✅ 几乎所有脚本 |
| L2 | 备用数据源切换 | ⚠️ 仅 AU_TIPS 等少数实现 |
| L3 | 缓存/历史值替代 | ❌ 基本未实现 |
| L4 | 数据库回补兜底 | ✅ common/db_utils 已提供，但仅约 4.5% 因子脚本调用 `save_l4_fallback()` |

核心问题：**`save_to_db()` 入库调用率高（251 个文件），但 L4 兜底函数 `save_l4_fallback()` 调用率极低（仅 12/266 ≈ 4.5%）**。这意味着一旦主数据源（L1）和备用源（L2）都失败，绝大多数脚本不会写入兜底记录，因子链会直接断裂。建议制定模板强制要求所有因子脚本在异常分支中调用 `save_l4_fallback()`。
