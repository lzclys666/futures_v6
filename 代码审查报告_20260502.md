## 二、P0 - 严重问题（立即修复，涉及交易安全）

### 1. CTP 账户密码硬编码

**文件**: `D:\futures_v6\run.py` 第 318-324 行
```python
CTP_PASSWORD = os.environ.get("VNPY_CTP_PASSWORD", "luzc19891222@")
```

**问题**: 交易密码以明文硬编码在源代码中，且默认值包含真实密码。代码提交到任何版本控制系统都会造成密码泄露。

**修复方案**: 
- 移除默认值，改为 `os.environ.get("VNPY_CTP_PASSWORD")` 
- 使用 `.env` 文件管理敏感信息
- 将 `.env` 加入 `.gitignore`

### 2. PaperBridge 盈亏计算逻辑错误

**文件**: `D:\futures_v6\services\vnpy_bridge.py` 第 1248-1256 行
```python
if offset == "OPEN":
    pos["volume"] += volume
    pos["price"] = price        # ← BUG: 平仓覆盖了开仓价
elif offset == "CLOSE" and pos["volume"] >= volume:
    pos["volume"] -= volume
```

**问题**: 
- 平仓时 (`CLOSE`) 执行了 `pos["price"] = price`（因为代码没有 `else` 分支，fallthrough到了 `if offset=="OPEN"` 块之后...不，这里实际上是 `if/elif` 结构）
- 再看一下... `if offset == "OPEN": ... pos["price"] = price`。平仓时不会进这个分支，所以 OK。但是 `cancel_order` 方法有个问题：

```python
def cancel_order(self, vt_orderid):
    return vt_orderid in self._orders  # 返回 bool，但调用处期望返回 str
```

**影响**: 撤单接口返回 `True/False` 而不是预期的撤单结果，前端可能无法正确处理。

### 3. VNpyBridge / PaperBridge 双单例冲突

**文件**: `D:\futures_v6\services\vnpy_bridge.py`

- `VNpyBridge` 使用单例模式（第 113-122 行）
- `PaperBridge` 在第 1382 行创建了全局实例 `bridge = PaperBridge()`
- `macro_api_server.py` 的 `get_vnpy_bridge()` 会尝试从 `services.vnpy_bridge import bridge`，但如果先导入的是 `PaperBridge` 实例，则无法获得真正的 VNpy 引擎连接

**问题**: 两个桥接实例可能共存，导致系统不确定当前使用的是哪一个，风控/持仓/账户数据来源混乱。

---

### 4. 风控规则体系四套不一致

**问题**: 系统中存在 **4 套风控规则定义**，内容互不一致：

| 来源 | 规则数量 | 规则ID命名 | 阈值 |
|------|---------|-----------|------|
| `core/risk/risk_engine.py` | 11条 (R1-R11) | `R10_MacroFuseRule` | 2%/5000元 |
| `routes/risk.py` | 12条 (含R12) | `R10_MACRO_CIRCUIT_BREAKER` | 3%/5000元 |
| `PaperBridge.get_risk_status()` | 12条 | `R10_MACRO_CIRCUIT_BREAKER`| -0.5 |
| `macro_api_server.py` get_risk_rules | 11条 | `"R10"` | `-0.5` |

**影响**: 
- 同一笔下单请求，不同接口返回不同的风控结果
- 前端不知道该信任哪个数据源
- 部分规则（如 R5 波动率过滤）在 `risk_engine.py` 中触发 BLOCK，在 `routes/risk.py` 中只触发 MEDIUM 警告

## 三、P1 - 高风险问题（本周内修复）

### 5. 风控引擎未接入实际交易流程

`RiskEngine` 虽然实现了完善的 11 条规则，但：
- `macro_demo_strategy.py` **完全没有集成风控引擎**
- `macro_risk_strategy.py` 集成了但未在 `run.py` 中加载使用
- `routes/risk.py` 的 `/api/risk/simulate` 自己重写了另一套风控检查，而非复用 `RiskEngine`

**后果**: 所有风控检查停留在"查看"阶段，实际下单时没有任何风控拦截。

---

### 6. 策略代码重复

`macro_demo_strategy.py` 和 `macro_risk_strategy.py` 高度重复：
- `SYMBOL_CONFIG` 字典完全复制（约 50 行）
- `extract_symbol()`、`get_symbol_config()`、`write_log_safe()` 方法一模一样
- `check_entry()` / `check_exit()` / `calc_tech_signal()` 逻辑几乎一致
- `load_macro_signal()` 完全复制

**修复方案**: 抽离基类 `MacroBaseStrategy`，让两个策略继承。

---

### 7. API 返回格式不一致

- `/api/macro/signal/all` 使用 `ApiResponse` 包装 → `{"code": 0, "message": "success", "data": [...]}`
- `/api/risk/status` 直接返回 `{"code": 0, "message": "success", "data": {...}}`
- `/api/risk/simulate` 也直接返回上述格式（缺少统一响应模型）

前端需要同时兼容两种格式，增加了维护成本。

### 10. `macro_scoring_engine.py` mock 数据误导

当 CSV 文件不存在时，引擎使用 Brownian motion 随机模拟生成信号分数。这在开发阶段可以接受，但在生产环境中：
- 用户/前端看到的"信号数据"实际上是随机数
- 历史打分数据（`get_score_history`）在没有 CSV 的日期也返回模拟值
- 前端没有明确的"模拟数据"标识

## 四、P2 - 中风险问题（两周内修复）

### 11. 代码组织问题

| 问题 | 文件 | 说明 |
|------|------|------|
| 主文件过长 | `macro_api_server.py` (723行) | 混合路由、模型、业务逻辑 |
| `except: pass` 过多 | 多处 | 大量异常被静默吞噬 |
| `print()` 混杂 | `run.py` | 生产代码不应使用 `print()` |
| 未使用导入 | `macro_api_server.py:30` | `from scipy.stats import pearsonr` 导入了从未使用 |

### 12. 单例模式的线程安全问题

`VNpyBridge` 的单例模式使用双重检查锁，但 `__init__` 方法不是线程安全的。`_cleanup()` 和 `stop()` 方法中有 `time.sleep()`，可能导致竞态条件。

### 13. 策略策略强平价格大幅偏离

`multi_factor_strategy.py` 第 386-388 行：
```python
if self.pos > 0:
    self.sell(price * 0.98, abs(self.pos))   # 以市价98%卖出
else:
    self.cover(price * 1.02, abs(self.pos))  # 以市价102%买入平仓
```

在实盘中，这可能导致不必要的滑点成本，高流动性品种下也无需如此大的偏离。

### 14. 日志体系混乱

- `run.py`: 使用 `print()` 
- `vnpy_bridge.py`: 使用 Python `logging` 模块
- 策略: 使用 VNpy 的 `write_log()`
- `macro_api_server.py`: 使用独立 logging logger

没有统一的日志级别、格式和存储策略。

## 五、P3 - 低风险问题（长期规划）

### 15. 架构文档与实际不一致
- `tech_arch_v6.md` 描述"进程1（VNpy执行层）+ 进程2（FastAPI宏观层）"，但 `run.py` 在无头模式下 `while True: sleep(1)` 阻塞了主线程
- 实际上两层无法同时运行，需要通过多进程或子线程方式分离

### 16. 文件编码问题
- `macro_scoring_engine.py` 第 1 行包含 UTF-8 BOM（EF BB BF），在某些编辑器中可能出现编码问题

### 17. 前端部分待完善
- `frontend/futures_trading/frontend/` 和 `macro_engine/frontend/` 两套前端代码共存，可能造成混乱
- 前端风控面板的后端API尚未完全实现（`PUT /api/risk/rules/{rule_id}` 实际未生效）

## 八、技术能力提升建议

1. **代码审查机制**：建议使用 GitHub Flow + PR Review，每人合入前至少一人Review
2. **测试先行**：核心模块（策略、风控、API）必须配套单元测试和集成测试
3. **配置管理**：所有环境差异用配置文件管理，禁止硬编码
4. **统一规范**：建议使用 pre-commit hook 做代码格式化（Black + isort + flake8）
5. **文档即代码**：架构文档与代码同步更新，接口变更先改文档再改代码
